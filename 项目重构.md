# EduFlowGraph 重构总结

本文档记录 AITutor 项目从旧版 `eduflowgraph/` 单体架构，重构为新版 `EduFlowGraph/` 模块化架构的完整过程、设计决策与实现细节。

---

## 一、为什么要重构

旧版系统有三个核心痛点：

### 1. Pipeline 过于臃肿

旧版 `eduflowgraph/pipeline.py` 约 **1700 行**，把以下职责全部揉在一起：

- 聊天与流式回复
- Episode 边界检测与抽取
- Concept / Skill 抽取与蒸馏
- 记忆图谱读写
- 检索与 Rerank
- 用户画像
- Dashboard 聚合

任何一个小改动都要在大文件里找逻辑，测试和维护成本都很高。

### 2. DataFlow 职责混乱

旧版用单一 `dataflow.jsonl` 同时记录：

- 原始对话事件（`user_message` / `assistant_message`）
- Episode 抽取完成事件
- Concept / Skill 状态变化
- 边界检测、验证等系统事件

结果是：**对话回放、记忆审计、Episode 边界判断** 都绑在同一条事件流上，查询困难，也不利于跨会话复用概念。

### 3. Concept 复用写死在代码里

旧版 `graph_store.py` 依赖 `CANONICAL_CONCEPT_GROUPS` 等硬编码分组（例如 Bayes / 概率相关概念），无法随着真实对话动态积累别名、跨会话合并相似概念。

---

## 二、重构目标

1. **拆分 DataFlow**：对话归对话，记忆状态归记忆状态。
2. **模块化 Pipeline**：按 store / memory / profile / orchestrator 分层。
3. **建立多维用户画像**：不是 DeepTutor 的 L1/L2/L3 通用分层，而是教育场景专用的 5 个维度。
4. **Concept 动态复用**：模糊匹配 + Embedding 相似度，去掉硬编码 canonical groups。
5. **接入新 Prompt**：使用 `EduMindFlow/ Prompt/` 中修订后的抽取 Prompt，并新增 Profile 抽取 Prompt。
6. **前端最后适配**：Next.js + Tailwind 工作台同步新 Schema 与新 API。

---

## 三、新架构总览

```
EduFlowGraph/
├── Prompt/                    # 全部 Prompt 模板（Markdown）
├── config.py                  # Settings（LLM / Embedding / Reranker / 路径）
├── schemas.py                 # Turn、MemoryEvent 等基础模型
├── prompts.py                 # Prompt 加载器
├── llm.py                     # 统一 LLM 客户端（chat / embedding / rerank）
├── skills.py                  # 教学动作 & 困难模式 taxonomy
├── pipeline.py                # 薄编排层 TutorPipeline
├── web_app.py                 # FastAPI 入口
│
├── store/                     # 存储层
│   ├── conversation_log.py    # 按 session 存 Turn
│   ├── memory_flow.py         # 记忆状态变更日志
│   ├── graph_store.py         # 概念 / Episode / Skill 图谱
│   └── profile_store.py       # 画像证据 + 物化视图
│
├── memory/                    # 记忆流水线
│   ├── buffer.py              # 缓冲 + Episode 边界检测
│   ├── episode_extractor.py   # Episode 抽取
│   ├── concept_extractor.py   # Concept 抽取 + 动态复用
│   ├── skill_pipeline.py      # Skill 证据 / 蒸馏 / 验证
│   └── retriever.py           # Concept → Episode → Skill 检索
│
└── profile/                   # 用户画像系统
    ├── dimensions.py          # 五维定义
    ├── extractor.py           # Turn / Episode 级证据抽取
    ├── aggregator.py          # 证据聚合与摘要
    └── retriever.py           # 画像感知检索 + RankFusion
```

数据流关系：

```
用户消息
  → TutorPipeline
      → ConversationLog（写 Turn）
      → BufferManager（累积待抽取片段）
      → MemoryRetriever + ProfileRetriever（检索增强）
      → LLM 回复
      → 边界检测 → Episode 抽取
          → Concept 抽取 → GraphStore
          → Skill 流水线 → GraphStore
          → Profile 抽取 → ProfileEvidenceStore → LearnerProfileStore
          → MemoryFlow（只记状态变化，不记原文）
```

## 四、DataFlow 拆分（核心改动之一）

### 4.1 ConversationLogs — `store/conversation_log.py`

**路径**：`data/conversations/{session_id}.jsonl`

**设计原则**：

- 每个 session 一个文件，append-only。
- **每一行是一个完整 Turn**，不是零散的 user/assistant event。
- 只存对话本身，不负责更新图谱或画像。

**Turn 结构**（`schemas.Turn`）：

```json
{
  "turn_index": 1,
  "timestamp": "2026-06-17T12:00:00+00:00",
  "session_id": "session_xxx",
  "user_message": "为什么不能直接用检测准确率当患病概率？",
  "assistant_message": "因为 P(A|B) 和 P(B|A) 方向不同……",
  "metadata": {
    "reasoning": "...",
    "usage": { "prompt_tokens": 200, "completion_tokens": 80 }
  }
}
```

**主要 API**：

| 方法 | 作用 |
|------|------|
| `append_turn()` | 追加一轮完整对话 |
| `list_turns()` / `iter_turns()` | 按 session 读取 |
| `session_messages()` | 转为 OpenAI 风格 messages |
| `list_sessions()` | 列出所有 session |
| `next_turn_index()` | 获取下一个 turn 序号 |
| `clear_all()` | 重置记忆时清空 |

Episode 边界检测直接从 Turn 序列读取，不再依赖混杂的 event 流。

### 4.2 MemoryFlow — `store/memory_flow.py`

**路径**：`data/memory_flow.jsonl`

**设计原则**：

- 单一 append-only 日志。
- **只记录记忆系统的状态变化**，不保存完整对话原文。
- payload 里只放引用：`session_id`、`turn_range`、`episode_id`、`concept_id` 等。

**支持的事件类型**：

| event_type | 含义 |
|------------|------|
| `episode_created` | 新 Episode 节点写入图谱 |
| `episode_extraction_failed` | Episode 抽取失败 |
| `concept_extracted` | Concept 抽取完成 |
| `concept_merged` | 两个 Concept 合并 |
| `concept_extraction_failed` | Concept 抽取失败 |
| `skill_evidence_added` | Skill 证据记录 |
| `skill_distilled` | Skill 蒸馏完成 |
| `skill_distillation_failed` | Skill 蒸馏失败 |
| `skill_validated` | Skill 验证结果 |
| `profile_updated` | 画像视图更新 |

**MemoryEvent 结构**（`schemas.MemoryEvent`）：

```json
{
  "event_id": "mf_20260617_120000_abc123",
  "timestamp": "...",
  "event_type": "episode_created",
  "session_id": "session_xxx",
  "payload": { "episode_id": "episode_...", "turn_range": [1, 4] }
}
```

这样可以从 MemoryFlow **重放记忆图谱演化过程**，而 ConversationLogs 专门负责 **对话回放**。

### 4.3 数据目录对照

| 旧版 | 新版 |
|------|------|
| `data/dataflow.jsonl`（混合） | `data/conversations/*.jsonl`（纯对话 Turn） |
| 同上 | `data/memory_flow.jsonl`（纯记忆状态） |
| `data/graph_nodes.json` | 保留 |
| `data/graph_edges.json` | 保留 |
| 无独立画像证据 | `data/profile_evidence.jsonl` |
| 无独立画像视图 | `data/learner_profile.json` |

> 当前实现说明（2026-06-23）：运行时主存储已切换为
> `data/eduflowgraph.db`。Conversation、MemoryFlow、Graph、Profile 和 Embedding
> 分别进入 SQLite 表，Embedding 使用独立 Float32 BLOB，不再重复写入节点 JSON
> 和 MemoryFlow。原 JSON/JSONL 文件仍保留为迁移来源和临时回滚入口。

旧数据迁移命令：

```bash
.venv/bin/python scripts/migrate_storage.py --data-dir data --dry-run
.venv/bin/python scripts/migrate_storage.py --data-dir data --apply
.venv/bin/python scripts/migrate_storage.py --data-dir data --verify
```

如果目标 SQLite 仅完成过空 schema 初始化，可在严格空表校验后使用
`--replace-empty`。迁移不会修改旧文件；观察期回滚可设置
`EDUFLOW_STORAGE_BACKEND=json`。如果 SQLite 已产生新数据，先执行：

```bash
.venv/bin/python scripts/export_storage.py \
  --database-path data/eduflowgraph.db \
  --output-dir data-export-$(date +%Y%m%d-%H%M%S)
```

验证导出后，再将 `EDUFLOW_DATA_DIR` 指向导出目录并切换 JSON backend。

## 五、Pipeline 模块化分解

旧版 ~1700 行单体 → 新版按职责拆成多个模块，`pipeline.py` 作为 **薄编排层**（约 1080 行，仍包含完整业务串联，但具体逻辑已下沉）。

### 5.1 `pipeline.py` — TutorPipeline 编排职责

**初始化时挂载的组件**：

| 组件 | 来源 |
|------|------|
| `ConversationLog` | store |
| `MemoryFlow` | store |
| `GraphStore` | store |
| `ProfileEvidenceStore` / `LearnerProfileStore` | store |
| `BufferManager` / `EpisodeBoundaryDetector` | memory/buffer |
| `HeuristicEpisodeExtractor` 等 | memory/* |
| `MemoryRetriever` | memory/retriever |
| `ProfileExtractor` / `ProfileRetriever` / `RankFusion` | profile/* |
| `LLMClient` | llm |

**核心公开方法**：

| 方法 | 说明 |
|------|------|
| `handle_user_message()` | 同步聊天 |
| `stream_user_message()` | SSE 流式聊天 |
| `force_extract()` | 手动触发当前 buffer 抽取 |
| `reset_memory()` | 清空对话、图谱、画像、MemoryFlow |
| `rebuild_retrieval_embeddings()` | 重建检索向量 |
| `dashboard()` | 返回前端工作台快照 |
| `refresh_from_storage()` | 从磁盘重载状态 |

**一轮对话的内部流程**（简化）：

1. `_start_tutor_turn()`：读历史 Turn → 检索记忆 + 画像 → 构造 Prompt
2. LLM 生成回答（同步或流式）
3. `_complete_tutor_turn()`：
   - 写入 `ConversationLog`
   - 更新 `BufferManager`
   - 调用边界检测；若应闭合 → `_extract_segment()`
4. `_extract_segment()` 链路：
   - Episode 抽取（启发式 + LLM fallback）
   - Concept 抽取 + 动态复用
   - Skill 证据 → 蒸馏 → 验证
   - Profile 证据抽取 → 聚合 → 更新视图
   - 各阶段向 `MemoryFlow` 写事件

### 5.2 Memory 流水线各模块

#### `memory/buffer.py`

- **`BufferManager`**：按 session 维护待抽取 Turn 缓冲；支持 `consume()` / `consume_prefix()`。
- **`EpisodeBoundaryDetector`**：从旧 `episode_detection.py` 迁移，使用新 Prompt `Episode_Detection_Prompt.md`；通过 LLM 判断当前 buffer 是否形成语义边界。

#### `memory/episode_extractor.py`

- 适配新 Prompt 输出格式。
- **新 Episode Schema**（扁平结构，替代旧 `learner_problem` / `tutor_action` / `learning_outcome`）：

```json
{
  "episode_type": "misconception_diagnosis",
  "title": "检测准确率与后验概率混淆",
  "summary": "学生混淆了 P(A|B) 和 P(B|A)",
  "learner": {
    "goal": "...",
    "obstacle": "...",
    "initial_state": "low",
    "evidence": ["..."]
  },
  "tutor": {
    "strategy": "contrastive_explanation",
    "key_moves": ["..."]
  },
  "outcome": {
    "status": "partial_success",
    "evidence": "...",
    "next_step": "..."
  },
  "memory_value": "..."
}
```

- 提供 `HeuristicEpisodeExtractor`（mock / 无 LLM 时的规则 fallback）、`coerce_episode_from_llm()`、`finalize_episode()`。

#### `memory/concept_extractor.py`

- 从新 Concept Prompt 解析 `role` / `learner_state` / `salience` / `evidence` 等边字段。
- 过滤 banned concept 名（如把教学动作误标为概念）。
- 与 `GraphStore` 配合做动态复用（见第六节）。

#### `memory/skill_pipeline.py`

- Skill 证据抽取 → 蒸馏 → 与已有 Skill 匹配 → 验证。
- 更新 teaching action taxonomy，新增/对齐：
  - `step_by_step_guidance`
  - `concrete_example`
  - `self_explanation_prompt`
  - `guided_practice`
  - `error_correction`
- 新增困难模式 `conceptual_confusion`。

#### `memory/retriever.py`

- 多阶段检索：Concept 召回 → Episode 扩展 → Skill 召回 → Rerank。
- 全部字段访问已适配新 Episode Schema（`learner` / `tutor` / `outcome`，顶层 `title` / `summary` 字符串）。

### 5.3 Profile 画像系统

#### 五维定义 — `profile/dimensions.py`

| 维度 | 中文含义 | 追踪内容 |
|------|----------|----------|
| `knowledge_mastery` | 知识掌握 | 各概念掌握度、混淆史 |
| `learning_style` | 学习风格 | 偏好举例 / 公式 / 分步等 |
| `difficulty_tendency` | 困难倾向 | 反复出现的困难模式 |
| `engagement_pattern` | 互动模式 | 提问深度、参与方式 |
| `growth_trajectory` | 成长轨迹 | 突破、平台期、进步信号 |

与 DeepTutor 的 L1/L2/L3 不同：这里是 **教育语义维度**，不是通用记忆分层。

#### 证据流

```
Turn / Episode
  → ProfileExtractor（LLM + mock fallback，Prompt: Profile_Extraction_Prompt.md）
  → ProfileEvidenceStore（append-only: profile_evidence.jsonl）
  → LearnerProfileStore.rebuild_from_evidence()
  → learner_profile.json（物化多维视图）
```

#### `store/profile_store.py` 要点

- 证据字段：`dimension`、`label`、`confidence`、`lifecycle`、`semantic_anchors`、`source_refs`。
- lifecycle：`active` / `reinforced` / `weak` / `retracted` / `resolved` / `expired`。
- Turn 级证据需达到支持阈值（`TURN_SUPPORT_THRESHOLD = 2`）才晋升到公开画像视图；Episode 级证据（`PROMOTED_STAGES = {"episode"}`）更容易固化。

#### `profile/retriever.py`

- `RankFusion`：把画像命中与 Concept/Episode/Skill 检索分融合。
- 通过 `semantic_anchors`（concept_names、teaching_actions、difficulty_patterns）做锚点加权。

## 六、Concept 动态复用（核心改动之二）

旧版硬编码 `CANONICAL_CONCEPT_GROUPS` → 新版 **`GraphStore` 动态匹配**。

### 匹配优先级

1. **精确名匹配**（大小写不敏感）
2. **别名匹配**（name 或 aliases 任一命中）
3. **归一化文本匹配**（去非字母数字、小写）
4. **Embedding 相似度**（cosine ≥ **0.92**，见 `find_similar_concept_by_embedding()`）

### 合并行为

- 命中已有 Concept → **累积 aliases**，更新 description / retrieval / updated_at。
- 发生 merge 时 → MemoryFlow 写 `concept_merged` 事件，便于审计。
- Concept 注册表持久化在 `graph_nodes.json`，**天然跨 session 复用**。

## 十二、与原 `eduflowgraph/` 的关系

- 新实现完整落在 **`EduFlowGraph/`**（注意大小写）。
- 旧目录 `eduflowgraph/` 仍保留部分历史文件，**新开发应以 `EduFlowGraph/` 为准**。

## 十三、原三个问题的落地答案

### Q1: DataFlow 太杂怎么办？

**答**：拆成 ConversationLogs（纯 Turn）+ MemoryFlow（纯状态变更）。对话回放、Episode 边界、记忆审计三条线解耦。

### Q2: 用户画像怎么构建？

**答**：Turn/Episode → ProfileExtractor → 证据账本 → 按五维聚合 → 物化视图；检索时通过 ProfileRetriever + RankFusion 参与记忆融合。

### Q3: Concept 如何跨对话复用？

**答**：GraphStore 动态 fuzzy + embedding 匹配，aliases 累积，merge 事件写入 MemoryFlow；去掉硬编码 canonical groups。
