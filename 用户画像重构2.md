# EduFlowGraph 用户画像模块重构方案：三模型画像 + Skill 适配机制

## 1. 重构目标

当前用户画像模块需要从“教学策略存储器”重构为“学习状态建模 + 情境感知 + Skill 个性化适配”的画像系统。

新的用户画像不再重复保存 Skill Node 中已有的教学程序，而是负责描述学生当前状态、当前学习情境，以及如何根据学生特点对检索到的 Skill 进行重排、过滤和轻量调节。

---

## 2. 核心设计原则

用户画像模块由三个子模型组成：

```text
UserProfile
├── LearnerModel
├── ContextModel
└── TeachingAdaptationModel
```

其中：

```text
LearnerModel：学生当前是怎样的学习者。
ContextModel：学生此刻处于什么学习场景、任务压力和状态约束中。
TeachingAdaptationModel：这个学生更适合如何选择、重排、过滤和调节 Skill。
```

注意：原来的 TeachingStrategyModel 不应继续作为“教学策略存储器”，建议改名为 TeachingAdaptationModel，避免与 Skill Node 冲突。

---

## 3. 三个模型的职责边界

### 3.1 LearnerModel：学习者模型

LearnerModel 用于描述学生当前的学习状态，包括学生在知识理解、推理方式、学习困难、薄弱点、认知水平上的表现。

它回答的问题是：

```text
这个学生现在懂什么？
不懂什么？
卡在哪里？
具备什么基础？
哪些掌握情况还需要验证？
```

LearnerModel 可以直接进入 Tutor Prompt，用于帮助 Tutor 理解学生当前状态。

---

### 3.2 ContextModel：情境模型

ContextModel 用于描述当前学习行为发生的场景，包括学生当前的学习阶段、情绪状态、任务目标、压力状态、对话意图等。

它回答的问题是：

```text
学生现在是在入门、理解、巩固、复习还是迁移？
学生当前是否焦虑、烦躁、积极或困惑？
学生是否有明确任务压力？
学生这轮对话是想快速解决问题，还是想深入理解？
```

ContextModel 可以直接进入 Tutor Prompt，用于帮助 Tutor 决定当前回答的节奏、深度和反馈方式。

---

### 3.3 TeachingAdaptationModel：教学适配模型

TeachingAdaptationModel 不保存具体教学程序，不重复 Skill Node 的职责。

它回答的问题是：

```text
对于这个学生，当前更适合调用哪些类型的 Skill？
哪些 Skill 应该升权？
哪些 Skill 应该降权或过滤？
同一个 Skill 在生成回答时应该如何轻量调整表达方式？
```

TeachingAdaptationModel 的主要作用是：

```text
1. 对召回的 Skill 进行重排。
2. 删除明显不适合当前学生或当前情境的 Skill。
3. 在没有合适 Skill 时，生成简短教学控制提示作为冷启动兜底。
4. 对最终选中的 Skill 做轻量表达调节。
```

TeachingAdaptationModel 不应作为完整文本直接注入 Prompt；它主要参与 Skill Recall 之后的 rerank/filter 流程。必要时只生成一句短控制提示进入 Prompt。

---

## 4. 与 Skill Node 的关系

Skill Node 的职责是保存可复用教学程序，例如：

```text
什么时候触发该 Skill？
Tutor 应该按什么步骤教学？
如何判断学生真正理解？
该 Skill 来自哪些成功 Episode？
```

TeachingAdaptationModel 的职责不是保存这些步骤，而是决定：

```text
这个学生当前更适合使用哪个 Skill？
这个 Skill 是否应该被降权？
是否应该先用低认知负荷版本？
是否应该避免公式密集型解释？
```

两者关系如下：

```text
Skill Node = 系统会什么教学方法。
TeachingAdaptationModel = 当前学生更适合用哪些教学方法。
```

---

## 5. 运行时流程

### 5.1 开启记忆增强模式时

当用户发起新问题后，系统执行：

```text
User Query
  ↓
Concept Retrieval
  ↓
Episode Retrieval
  ↓
Skill Retrieval
  ↓
TeachingAdaptationModel 对 Skill 重排/过滤
  ↓
构造 Personalized Tutor Context
  ↓
Tutor 生成回答
```

其中 Prompt 注入规则如下：

```text
LearnerModel：直接注入 Prompt。
ContextModel：直接注入 Prompt。
TeachingAdaptationModel：不直接整体注入，只参与 Skill 重排；必要时注入一句短控制提示。
Selected Skill：作为教学程序注入 Prompt。
```

---

## 6. Skill 稀疏情况下的冷启动策略

前期 Skill Node 数量很少是正常现象，因为 Skill 提取应该保持严格，不应为了数量降低质量。

系统必须支持 Skill 缺失时的 graceful fallback。

### 6.1 有合适 Skill 时

```text
使用 Concept + Episode + LearnerModel + ContextModel + Selected Skill 生成回答。
```

### 6.2 没有合适 Skill 时

```text
跳过 Skill Block。
使用 Concept + Episode + LearnerModel + ContextModel。
同时让 TeachingAdaptationModel 生成一句短教学控制提示。
```

例如：

```text
本轮回答应先用直观类比建立理解，再逐步进入形式化推导，避免一开始堆叠复杂公式。
```

该控制提示可以进入 Prompt，但不要把完整 TeachingAdaptationModel 画像原文塞进 Prompt。

---

## 7. Skill 重排建议

Skill Recall 后，建议根据以下信号重排：

```text
base_skill_score：Skill 与当前 query 的语义匹配程度。
episode_link_score：Skill 关联 Episode 与当前问题的相关性。
skill_confidence：Skill 自身置信度。
adaptation_match_score：Skill 是否符合 TeachingAdaptationModel 中的个性化适配倾向。
context_fit_score：Skill 是否适合当前 ContextModel。
negative_penalty：Skill 是否触发不适合当前学生的负向偏好。
```

可以使用简单加权方式：

```text
final_skill_score =
  0.35 * base_skill_score
+ 0.20 * episode_link_score
+ 0.15 * skill_confidence
+ 0.20 * adaptation_match_score
+ 0.10 * context_fit_score
- negative_penalty
```
最多保留 Top k（初始化 k = 4） Skills。

## 8. 用两个示例对话得到的合理画像内容

根据“椭圆面积公式推导”和“PPO 算法学习”两个对话，合理画像如下。

### 8.1 LearnerModel 示例

```text
学生具备一定微积分基础，能够接受定积分、三角换元等形式化推导；在椭圆面积公式学习中，主要困难不是记忆公式本身，而是理解公式来源与推导路径。学生在强化学习主题上属于初学者，对 PPO 等复杂算法的术语和公式较容易产生认知负荷，但在获得直观解释后愿意继续进入系统学习和公式推导。当前多处掌握状态仍属于初步理解，需要通过复述、迁移题或简单推导任务进一步验证。
```

### 8.2 ContextModel 示例

```text
学生在椭圆面积学习中最初处于明显受挫状态，随后通过微积分推导获得理解感；在 PPO 学习中，学生先被复杂公式压住，经过生活化比喻后转为积极状态，并表达出系统学习强化学习和深入公式推导的意愿。当前整体处于从直觉理解转向系统学习与形式化推导的阶段，无明显外部任务压力。
```

### 8.3 TeachingAdaptationModel 示例

```text
该学生适合“直觉引入 → 结构化拆解 → 形式化推导 → 迁移验证”的教学节奏。面对全新复杂概念或算法时，应优先选择类比引入、核心思想压缩、低认知负荷解释类 Skill，避免一开始直接堆叠完整理论和复杂公式；当学生表示已经理解基本直觉或主动要求公式推导时，应切换到分步推导、公式拆解、worked example、自我复述检查等 Skill。对于数学公式类问题，学生并不排斥严谨推导，反而在具备相关基础时更偏好用微积分或形式化方法理解公式来源。
```

注意：TeachingAdaptationModel 不应保存“椭圆面积如何推导”或“PPO 公式如何推导”的具体步骤，这些应由 Skill Node 或 Tutor 实时生成承担。

## 9. UI 展示建议

用户画像页面可以保留三个卡片：

```text
学习者模型
情境模型
教学适配模型
```

其中第三个卡片不要再叫“教学策略模型”，建议改为：

```text
教学适配模型
```

其说明文案为：

```text
根据学生历史学习表现，判断哪些教学 Skill 更适合当前学生，并用于 Skill 的重排、过滤和轻量调节。
```

---

## 10. 代码重构任务

请按以下方向重构代码：

1. 将旧的 TeachingStrategyModel 重命名或改造为 TeachingAdaptationModel。
2. LearnerModel 和 ContextModel 继续作为 Prompt 注入上下文。
3. TeachingAdaptationModel 不再作为完整 Prompt Block 直接注入。
4. 在 Skill Recall 后新增 adaptation rerank/filter 流程。
5. 如果无合适 Skill，则由 TeachingAdaptationModel 生成一句短教学控制提示作为 fallback。
6. 更新用户画像 UI，将“教学策略模型”改为“教学适配模型”。
7. 更新画像抽取 Prompt，使其区分：

   * 学生状态信息进入 LearnerModel；
   * 当前学习场景进入 ContextModel；
   * Skill 选择偏好、升降权规则、避免项进入 TeachingAdaptationModel。
8. 保持 Skill Node 严格蒸馏，不要为了冷启动放宽 Skill 生成标准。
9. 支持 Seed Skill / Candidate Skill / Active Skill 三种状态，但低置信度 Skill 不应强制注入。
10. Skill 缺失时系统必须正常回答，不应报错或生成空上下文。

---

## 11. 最终原则

```text
LearnerModel 负责理解学生。
ContextModel 负责理解当前学习场景。
TeachingAdaptationModel 负责个性化调度 Skill。
Skill Node 负责保存可复用教学程序。
Episode Node 负责提供真实学习证据。
Concept Node 负责定位当前学习内容。
```

最终目标不是让用户画像重复保存教学方法，而是让用户画像帮助系统更好地选择、调整和使用已有 Skill，从而实现更稳定、更可解释的个性化辅导。
